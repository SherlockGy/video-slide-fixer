# Video Slide Fixer

从PPT风格视频中提取幻灯片、检查质量问题、通过 Gemini API 重新生成修复版图片。

---

## 一、前置检查

### 1.1 工具可用性

本技能的 `scripts/` 目录已内置所有工具：
```
scripts/ffmpeg.exe      ← 视频处理
scripts/ffprobe.exe     ← 视频元数据分析
scripts/fend.exe        ← 精确数学计算（帧号计算等）
scripts/slides-fix.exe  ← 幻灯片图片修复（Go 语言开发）
scripts/model.conf      ← Gemini 模型配置文件
```

Claude Code 加载技能时会在系统提示中给出基础目录路径（"Base directory for this skill: ..."）。使用时直接拼接完整路径调用，例如：
```
"<基础目录>/scripts/ffmpeg.exe" -version
"<基础目录>/scripts/slides-fix.exe" -batch tasks.json
```
注意路径需用引号包裹。后续文档中 `ffmpeg`、`ffprobe`、`fend`、`slides-fix.exe` 均指 `scripts/` 下的对应工具，实际调用时替换为完整路径。

使用前需在项目工作目录下创建 `.env` 文件并填入 Gemini API 密钥（可从 `tool-source/slides-fix/.env.example` 复制）：
```
GEMINI_API_KEY=你的密钥
```

> 源码位于 `tool-source/slides-fix/`，可随时修改、编译后替换 `slides-fix.exe`。详见 `tool-source/slides-fix/README.md`。

### 1.2 视频元数据分析
```bash
ffprobe -v quiet -print_format json -show_format -show_streams "<视频路径>"
```
记录关键信息：时长、分辨率、帧率、编码格式、音频参数。

### 1.3 计算 Gemini 目标输出分辨率

从 ffprobe 获取源视频的宽（W）和高（H），按**同比例放大到长边 2048px** 计算目标分辨率：

```
若 W ≥ H（横屏）：目标宽 = 2048，目标高 = round(H × 2048 / W)
若 H > W（竖屏）：目标高 = 2048，目标宽 = round(W × 2048 / H)
```

**示例**：
| 源分辨率 | 比例 | 目标分辨率 |
|----------|------|------------|
| 1280×720 | 16:9 | 2048×1152 |
| 1920×1080 | 16:9 | 2048×1152 |
| 1080×1920 | 9:16 | 1152×2048 |
| 1280×960 | 4:3 | 2048×1536 |

计算后将目标分辨率记为 `{TARGET_W}×{TARGET_H}`，后续所有 Gemini 提示词中统一使用该值。

### 1.4 音频分离与文案获取

#### 1.4.1 分离 MP3

在获取视频元数据后，立即提取音频文件供用户转写文案：

```bash
ffmpeg -i "<视频路径>" -vn -acodec libmp3lame -q:a 2 "<项目目录>/<视频名>.mp3"
```

#### 1.4.2 等待用户提供文案

提取完成后，**暂停流程**，提示用户：

> MP3 已提取至 `<路径>`。请将音频转写为文案后放到项目目录下（支持 `.txt` 或 `.md` 格式），然后告诉我继续。

**暂停等待文案期间，可先继续执行第二节幻灯片提取**（场景检测、帧提取等不依赖文案）。文案就绪后再进入第三节"视觉质量检查（图文对照）"。

#### 1.4.3 读取文案

用户确认后，用 Read 工具读取文案文件，通读全文，建立对视频叙事脉络的理解。注意文案中的：
- 章节/段落分界（通常对应幻灯片切换）
- 关键术语和专有名词（后续检查文字正确性的权威依据）
- 叙述节奏（哪些段落对应纯插图/动画过渡，哪些对应文字密集的幻灯片）

## 二、幻灯片提取

### 2.1 场景检测（获取时间戳）

```bash
ffmpeg -i "<视频路径>" \
  -vf "select='gt(scene,<阈值>)',showinfo" \
  -vsync vfr -f null - 2>&1 | grep "pts_time"
```

**阈值选择指南**：
| 视频类型 | 推荐阈值 | 说明 |
|----------|----------|------|
| 纯PPT切换（无动画） | 0.3 - 0.4 | 场景变化大，高阈值避免误检 |
| 带动画的PPT（NotebookLM等） | 0.10 - 0.20 | 动画使场景变化平缓，需低阈值 |
| 混合内容视频 | 0.2 - 0.3 | 中间值平衡 |

**调参策略**：从 0.3 开始，若结果数量明显少于预期幻灯片数则降低；若结果过多（含大量动画中间帧）则升高。可多个阈值并行测试对比。

**提取时间戳的 grep 兼容性**：Windows Git Bash 下 `grep -P`（Perl正则）不可用，改用 sed：
```bash
ffmpeg -i "<视频>" -vf "select='gt(scene,0.15)',showinfo" -vsync vfr -f null - 2>&1 \
  | grep "pts_time" | sed 's/.*pts_time:\([0-9.]*\).*/\1/'
```

### 2.2 按时间戳批量提取帧

对每个场景切换时间点加 **+3秒偏移**（等动画渲染完毕），提取该时刻的画面：

```bash
TIMES=(
  "<时间戳1>:<文件名1>"
  "<时间戳2>:<文件名2>"
  # ...
)

for entry in "${TIMES[@]}"; do
  IFS=':' read -r ts name <<< "$entry"
  ffmpeg -y -ss "$ts" \
    -i "<视频路径>" \
    -frames:v 1 \
    "<输出目录>/${name}.png"
done
```

**命名规范**：`slide_NNN_MMmSSs.png`（如 `slide_001_00m03s.png`）

**关键参数**：
- `-ss` 放在 `-i` 之前：快速输入定位（seek to keyframe）
- `-frames:v 1`：只取一帧
- 输出为 PNG 格式：无损压缩，不丢失任何像素信息

### 2.3 补充提取（长间隔场景）

对场景间隔 >40秒的片段，在中间额外提取1帧，防止遗漏渐进式动画内容。命名加 `b` 后缀如 `slide_007b_03m00s.png`。

> **实测经验**：补充帧经常与主帧内容完全一致（长间隔并不意味着有新内容，可能只是动画过渡较慢）。这些重复帧可在检查阶段标记忽略，无需删除。

### 2.4 Windows 路径注意事项

- ffmpeg 支持 `C:/path/to/file` 格式（正斜杠）
- 在 Git Bash 中，`/tmp` 映射到 `$TEMP`（AppData/Local/Temp）
- Read 工具需要 Windows 格式路径：`C:\Users\...\file.png`
- 用 `cygpath -w <路径>` 互转

## 三、视觉质量检查（图文对照）

### 3.0 建立图文对照关系

在逐帧检查之前，先根据文案和时间戳建立**幻灯片-文案段落映射**：

1. 按幻灯片时间戳顺序，将文案按叙述脉络切分为段落
2. 每张幻灯片对应文案中正在讲述的段落（时间上下文匹配）
3. 输出映射表：
```
| 幻灯片 | 时间点 | 对应文案段落（摘要） |
```

这个映射表是后续所有检查和修复的核心依据：
- **文字纠错**：以文案为权威来源，判断画面中的文字是否正确
- **纯图片补文字**：根据对应段落的内容决定是否需要、以及补什么文字

### 3.1 逐帧读取与检查

用 Read 工具逐张读取 slides_all/ 中的图片（Read 支持 JPG/PNG），**结合对应文案段落**，按以下清单检查：

> **核心判断原则**：质量检查的重点是**逐字审视汉字的扭曲、破损、笔画不完整**，尤其是表格内的中小字体。不要被整体画面的"看起来还行"所迷惑——AI 生成的画面整体观感往往不错，但放大或仔细看单个汉字时，扭曲、缺笔、变形的问题才会暴露。**小字比大字更容易出问题**。

| 检查项 | PASS 条件 | FAIL 特征 |
|--------|-----------|-----------|
| **中文字符完整性（最重要）** | 逐字检查：每个汉字笔画完整、结构正确、无扭曲 | 单字级别的笔画缺失、扭曲变形、融化模糊、像素化。**尤其关注表格中的小字**——小字号的问题率远高于大标题 |
| **边缘异形** | 边缘/角落干净无多余元素 | 出现不明符号、半截文字、颜色溢出 |
| **内容完整性** | 文字/图表完整 | 文字截断、图表缺失元素 |
| **整体一致性** | 风格与其他幻灯片一致 | 配色异常、布局错乱 |
| **图文一致性** | 画面文字与文案内容吻合 | 画面文字与文案描述矛盾或缺失关键信息 |
| **语言一致性** | 文案为中文时，画面文字也应为中文 | 出现英文而文案对应内容为中文（需翻译替换） |

**AI生成视频的典型文字问题模式**（实测总结）：
| 模式 | 示例 | 说明 |
|------|------|------|
| **形近字替换** | "最优解"→"毬拔解"，"顶层设计"→"顶厝设计" | AI生成的字看起来像但实际是错字 |
| **完全乱码** | "汝小角"、"谓暑设计"、"计标芳会" | 边缘/角落区域常出现无意义的字符组合 |
| **文字拆分错乱** | "参谋长"→"参兵谋长"，"供应链"→"供应售链" | 每个字本身没有错，但插入了多余的字或顺序错乱，导致词语不通。以文案为准纠正 |
| **英文未翻译** | "PASSIVE CHOICE"→应为"被动选择"，"ACTIVE DESIGN"→应为"主动设计" | AI生成时使用了英文，但文案为中文，需替换为对应中文 |
| **多处聚集** | 同一张幻灯片上多处文字同时扭曲 | 一旦一帧有问题，通常不止一处 |
| **边缘高发** | 问题集中在画面边缘而非中心 | 中心区域的大号标题通常正确 |

**实测参考**：NotebookLM 生成的 ~11分钟视频，29帧中5帧有问题（约17%故障率）。

### 3.2 纯图片帧的文字补充决策

对于**完全没有文字**或**只有极少装饰性文字**的插图帧，结合对应文案段落判断是否需要补充文字：

**判断标准**：
| 场景 | 决策 | 理由 |
|------|------|------|
| 插图已能自解释，文案只是旁白补充 | **不补** | 强行加字反而破坏画面 |
| 插图含义模糊，文案中有精炼的关键词/短语可点题 | **补** | 一两个词让画面更有信息量 |
| 连续多帧纯插图，观众可能失去上下文 | **酌情补** | 在关键帧加一句话提示脉络即可 |
| 章节过渡/标题页已有标题 | **不补** | 已经有文字了 |

**补充文字的风格要求**：
- **少即是多**：最多一行短句或几个关键词，不要长段落
- **口语化、自然**：像"讲给你听"的语气，避免公文/教科书式的生硬表达。例如用"各干各的，效率能高吗？"而非"缺乏协同导致效率低下"
- **与整体视频风格一致**：观察视频中已有文字帧的表达风格（比喻、反问、口语化程度），补充文字应保持同样的调性
- **位置合理**：文字应放在画面留白处，不遮挡核心插图元素
- **只补大/中字体，不要主动添加小字体**：补充的文字应以醒目的大字体或清晰的中字体呈现，不要添加小字号的注释、标注或说明文字
- **不要画蛇添足**：如果拿不准，宁可不补。宁缺毋滥

**记录格式**：对需要补充文字的帧，在检查记录表的问题描述中注明"建议补充文字：XXX"。

### 3.3 建立检查记录表

对每张幻灯片输出一行记录：
```
| 文件名 | 对应文案 | 主要内容(简述) | 文字完整性 | 边缘质量 | 完整性 | 判定 | 问题描述 |
```

判定类型：
- **PASS**：无需处理
- **FAIL-文字**：文字扭曲/乱码/形近字，需修复
- **FAIL-补字**：纯图片帧，建议补充文字（附建议内容）
- **SKIP**：重复帧或片尾页，跳过

**高效检查方法**：每次用 Read 工具并行读取 6-8 张图片（Read 支持多图并行），一批审查后记录结果，再读下一批。29帧约4批即可完成。

### 3.4 标记问题帧

将 FAIL（含 FAIL-文字和 FAIL-补字）的帧复制到 slides_issues/：
```bash
cp "<slides_all/问题文件.png>" "<slides_issues/>"
```

## 四、Gemini API 重新生成（slides-fix 工具）

### 4.1 准备工作

1. **确认 .env 文件**已在项目工作目录中配置好 `GEMINI_API_KEY`
2. **确认 slides_issues/ 目录**中已放置需要修复的问题帧
3. **创建 slides_fixed/ 目录**用于存放输出

### 4.2 编写 tasks.json

在项目工作目录下创建 `tasks.json`，定义批量任务：

```json
[
  {
    "image": "slides_issues/slide_010_03m48s.png",
    "output": "slides_fixed/slide_010_fixed.png",
    "prompt": "\"毬拔解\"改为\"最优解\"，其余不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除"
  },
  {
    "image": "slides_issues/slide_018_07m18s.png",
    "output": "slides_fixed/slide_018_fixed.png",
    "prompt": "重绘汉字部分，确保没有扭曲、破损，边缘乱码删除，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除"
  }
]
```

**提示词编写要点**：
- 只描述要改什么（diff 风格），不要复述整张画面内容
- 指出错误文字及正确内容
- 附上"输出分辨率为 {TARGET_W}×{TARGET_H}"（1.3 节计算得出的目标分辨率）
- 参考第六节的提示词模板

### 4.3 运行批量修复

```bash
# 使用技能自带的预编译工具
slides-fix.exe -batch tasks.json -delay 10

# 或指定不同模型和并发数
slides-fix.exe -batch tasks.json -model gemini-3.1-flash-image-preview -delay 15 -concurrency 3
```

### 4.4 单张修复（调试/测试用）

```bash
slides-fix.exe \
  -image slides_issues/slide_010_03m48s.png \
  -prompt "请根据附图重新生成..." \
  -output slides_fixed/slide_010_fixed.png
```

### 4.5 处理失败任务

批量运行结束后会报告成功/失败数量。对失败的任务：

1. **网络错误（unexpected EOF）**：通常是临时性问题，创建仅含失败任务的 `tasks_retry.json`，重新运行
2. **429 / RESOURCE_EXHAUSTED**：已触发限流，工具会自动等待 60s 后重试；若仍失败，增大 `-delay` 参数或等待一段时间后重试
3. **无图片返回（text only）**：模型可能不支持该提示词的图片生成，调整提示词后重试

### 4.6 验证修复质量

用 Read 工具逐张查看 slides_fixed/ 中的图片，确认：
- 问题文字已修正
- 整体风格与原始幻灯片一致
- 未引入新的错误

若结果不满意，调整提示词后重新运行单张修复。

### 4.7 Gemini 网页版修复（表格密集文字场景，Playwright MCP）

**适用场景**：图片中包含**表格且有较多汉字**，文字存在扭曲、变形。Gemini 网页版对此类场景的文字重绘质量**显著优于 API**。

**触发时机**：在第三节质量检查中发现表格密集文字问题帧时，检测当前环境是否有可用的 Playwright MCP 工具（`browser_navigate` 等）。若可用，向用户询问：

> 检测到 {N} 张表格/文字密集的问题帧。当前环境有 Playwright MCP 可用，Gemini 网页版对此类场景的文字重绘效果显著优于 API。
> 是否使用 Gemini 网页版修复？
>
> **使用前请确认**：
> 1. 已在浏览器中登录 Google 账号（gemini.google.com 可正常访问）
> 2. 已关闭浏览器"下载前询问每个文件的保存位置"设置（Chrome: 设置 → 下载内容 → 关闭该选项），否则每次下载都会弹出系统对话框需要手动点击，Playwright 无法控制系统原生对话框

用户确认后，按 [references/gemini-web-playwright.md](references/gemini-web-playwright.md) 中的完整流程逐张处理。

## 六、提示词模板

### 提示词编写原则

**核心原则——diff 风格，只说要改什么**：
Gemini 已经能看到附图，所以提示词**只描述要改什么**，不要复述画面内容。越简洁，模型越不容易偏离原图。

1. **文字正确性（最高优先级）**：指出哪些文字有问题、改成什么。如果整体文字模糊，一句"重绘汉字部分，确保没有扭曲、破损"即可，不需要逐字列出画面上的所有文字。
   - **文字拆分错乱也要纠正**：如"参兵谋长"→"参谋长"，每个字本身没错但词语不通，同样需要指出并替换为正确文字。
   - **英文翻译为中文**：当文案为中文时，画面中的英文应替换为对应中文。如"PASSIVE CHOICE"→"被动选择"、"ACTIVE DESIGN"→"主动设计"。以文案用语为准。
2. **画面元素尽量不变**：不需要改的部分说"XX不变"即可，不要展开描述。不要重新设计画面。
3. **边缘异型字默认删除**：边缘乱码直接说"边缘乱码删除"。
4. **指定输出分辨率**：固定附上"输出分辨率为 {TARGET_W}×{TARGET_H}"（1.3 节计算得出），确保 Gemini 输出统一尺寸且比例与源视频一致。
5. **右下角标识处理**：有 NotebookLM 标识时加"右下角标识清除"。没有标识时完全不提右下角。
   - 不要用"移除logo/水印"——触发 Gemini 去水印策略
   - 不要用"右下角区域保持空白干净"——模型可能误解，反而造成画面异常
6. **字体大小与清晰度要求**：
   - **大字体**（标题、核心关键词）：使用符合画面整体风格的字体，可以有设计感
   - **中字体**（正文、要点列表）：**必须使用最清晰的字体，边缘锐利**，可读性优先于美观
   - **小字体**：**不要主动添加**。不要生成注释、脚注、标注等小字号文字。如果原图已有小字体且需要修复，保持原有大小即可
   - 在提示词中对应表述：大字体加"字体风格与画面一致"，中字体加"字体清晰锐利"

**提示词写法**：
- 用简短动作词："去掉""放大""替换为""不变""清除""重绘"
- 示例（局部修改）：`把左边皇冠下面的钻石去掉，皇冠放大，指向右边，右边不变，右下角标识清除`
- 示例（文字修复）：`重绘汉字部分，确保没有扭曲、破损，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除`
- 示例（特定纠错）：`右侧"毬拔解"改为"最优解"，其余不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除`
- 只有在画面严重损坏需要完整重建时（模板D），才需要详细描述内容

### 模板A：整体文字重绘（最常用）

适用：画面中文字整体模糊/扭曲，不需要逐个指出。

```
重绘汉字部分，确保没有扭曲、破损，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

可选追加：如果边缘有明显乱码，加"边缘乱码删除"。

### 模板B：特定文字纠错

适用：个别文字被替换为形近字、文字拆分错乱、或英文需翻译为中文。

```
"毬拔解"改为"最优解"，其余不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

文字拆分错乱（字没错但词不通）：
```
"参兵谋长"改为"参谋长"，其余不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

英文翻译为中文：
```
"PASSIVE CHOICE"改为"被动选择"，"ACTIVE DESIGN"改为"主动设计"，其余不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

多处纠错时逐个列出：
```
"毬拔解"改为"最优解"，"谓暑设计"改为"顶层设计"，边缘乱码删除，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

### 模板C：画面元素修改

适用：需要调整画面中的非文字元素（图标、插画、布局等）。

```
把左边皇冠下面的钻石去掉，皇冠放大，指向右边，右边不变，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

要点：只说要改什么，不变的部分说"XX不变"，不要展开描述。

### 模板D：完整重建（原图损坏严重时）

适用：画面多处严重损坏，diff 修复不如全部重建。**这是唯一需要详细描述内容的场景。**

```
请根据附图重新生成一张PPT风格的幻灯片图片。
重新绘制所有中文字符，确保笔画完整、无乱码。
输出分辨率为 {TARGET_W}×{TARGET_H}。
布局尽量保持与原图一致。
右下角标识清除。

画面中应出现的文字内容：
{需要列出的文字，仅当原图损坏到无法辨认时才需要}
```

### 模板E：纯图片帧补充文字

适用：画面为纯插图/动画，无文字或文字极少，根据文案需要补充点题文字。

```
在画面{位置}添加文字"{补充内容}"，字体风格与视频其他文字帧保持一致，不遮挡主体插图，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

**要点**：
- 补充文字从文案中提炼，用口语化、自然的表达，不要教科书腔
- 指定文字放置位置（顶部/底部/左侧留白处等）
- 明确要求字体风格与已有文字帧一致
- 文字量克制：一行短句或几个关键词足矣

**示例**：
```
在画面底部添加文字"各划各的桨，船能走直吗？"，字体风格与视频其他文字帧一致，不遮挡船和人物，输出分辨率为 2048×1152，右下角标识清除
```

### 组合用法

提示词可自由组合，保持简洁即可：
```
重绘汉字部分，确保没有扭曲、破损，"谓暑设计"改为"顶层设计"，边缘乱码删除，输出分辨率为 {TARGET_W}×{TARGET_H}，右下角标识清除
```

> **注意**：原图有 NotebookLM 标识时加"右下角标识清除"。原图没有标识时完全不提右下角。

## 七、完整目录结构模板

```
<项目目录>/
├── <源视频>.mp4
├── <源视频>.mp3           ← 分离的音频（供用户转写文案）
├── <文案>.txt 或 .md      ← 用户提供的视频文案（音频转写）
├── slides_all/            ← 全量幻灯片截图 (slide_NNN_MMmSSs.png)
├── slides_issues/         ← 有问题的帧（从 slides_all 复制）
├── slides_fixed/          ← Gemini 重新生成的修复版
└── slides-fix/            ← slides-fix 工具工作目录（可选，用于就地运行）
    ├── .env               ← Gemini API 密钥
    ├── tasks.json         ← 批量任务配置
    └── slides-fix.exe     ← 可直接复制自技能 scripts/ 目录
```

## 九、执行检查清单

```
□ 1. 确认技能基础目录路径，后续命令使用完整路径调用 scripts/ 下的工具
□ 2. 配置 .env（填入 Gemini API 密钥）
□ 3. 获取视频元数据（ffprobe）
□ 3b. 根据源视频宽高计算 Gemini 目标输出分辨率（长边 2048px，等比缩放）
□ 3c. 分离 MP3 音频
□ 3d. ⏸ 暂停，等待用户提供文案文件（.txt / .md）
□ 3e. 读取文案，通读全文，理解叙事脉络
□ 4. 创建三个输出目录：slides_all/ slides_issues/ slides_fixed/
□ 5. 场景检测获取时间戳（ffmpeg + scene filter）
□ 6. 计算提取时间点（场景切换 +3s 偏移）
□ 7. 批量提取帧（主帧 + 长间隔补充帧）
□ 8. 建立幻灯片-文案段落映射表
□ 9. 逐帧视觉检查 + 图文对照（Read 工具，每批 6-8 张并行）
□ 9b. 纯图片帧补文字决策（结合文案，风格自然不生硬，宁缺毋滥）
□ 10. 标记问题帧并复制到 slides_issues/
□ 11. 输出问题汇总表（文件名、对应文案、问题类型、具体描述、正确文字/建议补充文字）
□ 11b. 若存在表格密集文字问题帧 → 检测 Playwright MCP 是否可用 → 若可用，询问用户是否使用 Gemini 网页版修复（提醒预先登录 + 关闭下载询问设置）→ 详见 4.7 节
□ 12. 编写 tasks.json（定义批量修复任务，提示词用 diff 风格，补字用模板E）；网页版修复的帧不需要加入 tasks.json
□ 13. 运行 slides-fix.exe -batch tasks.json（跳过已由网页版修复的帧）
□ 14. 处理失败任务（重试或调整提示词）
□ 15. 验证修复质量（Read 工具逐张检查 slides_fixed/）
□ 16. 计算帧范围（场景时间戳 × fps → 起始帧/结束帧）
□ 17. 展示帧替换表，等待用户确认
□ 18. 执行 ffmpeg overlay 回写视频（输出独立文件 *_fixed.mp4）
□ 19. 抽查输出视频中替换区间的画面
□ 20. （NotebookLM 视频）去除片尾：用 -t 截断到最后一张内容幻灯片的结束时间戳，必须重编码（-c:v libx264），不能用 -c:v copy
```
